## Appendix A
## Appendix A: API Endpoints and Integration

### Repository and Configuration

Repository: `https://github.com/To1enoff/RiskEdu`  
Local API Base: `http://localhost:3000`  
Swagger UI: `GET /docs`

Authentication model:
- Protected routes require `Authorization: Bearer <token>`.
- Invalid or missing tokens return `401 Unauthorized`.
- Role-restricted routes return `403 Forbidden`.
- Missing entities return `404 Not Found`.
- Validation errors return `400 Bad Request`.

The platform follows a service-oriented integration model:

`Frontend -> Backend API -> PostgreSQL / ML Service -> Response`

The backend acts as the orchestration layer. It validates user input, persists academic records, invokes the ML service for prediction and explainability, applies domain rules such as auto-fail constraints, and returns aggregated results to the frontend.

### Integration Summary

The core operational sequence is:

1. The user enters academic data in the frontend.
2. The frontend sends requests to the NestJS backend.
3. The backend validates the payload, applies RBAC and academic rules, and writes course data to PostgreSQL.
4. The backend prepares model features and calls the FastAPI ML service.
5. The ML service runs inference and explainability routines against the trained model.
6. The backend combines statistical prediction, hard fail rules, and suggestion logic.
7. The frontend receives the final result and renders risk, explainability, and recommendations.

The corresponding system diagrams are provided in `docs/architecture-diagrams.md`.

## Appendix B: Environment, Deployment, and Supplemental Evidence

### Environment and Deployment Configuration

RiskEdu is implemented as a four-service monorepo:

- `frontend`: React + TypeScript + Vite + Tailwind CSS
- `backend`: NestJS + TypeORM + JWT/RBAC
- `ml-service`: FastAPI + scikit-learn + SHAP
- `postgres`: PostgreSQL persistence layer

Local development is orchestrated through Docker Compose:

```bash
npm run dev
```

Primary local endpoints:

- Frontend: `http://localhost:5173`
- Backend: `http://localhost:3000`
- Swagger: `http://localhost:3000/docs`
- ML service: `http://localhost:8000`

Required environment variables are split by service:

**Backend**
- `PORT`
- `DATABASE_URL`
- `ML_SERVICE_URL`
- `JWT_SECRET`
- `JWT_EXPIRES_IN`
- `CORS_ORIGIN`

- optional Gemini configuration for AI suggestions and syllabus parsing

**Frontend**
- `VITE_API_URL`

**ML Service**
- `DATA_ROOT`
- `ARTIFACT_DIR`
- `TRAIN_DATASET`
- `RANDOM_STATE`

Production deployment configuration:

- Frontend: GitHub Pages
- Backend: Render Web Service
- ML service: Render Web Service
- Database: Render PostgreSQL

### Academic and Operational Rules

The deployed system enforces domain-specific academic logic:

- Course duration is fixed to `15 weeks`.
- Course component weights must sum to `100`.
- The current week is auto-derived from semester start rules.
- Future weeks are not counted into grade aggregation.
- Empty past weekly fields are treated as missing records, not implicit zeroes.
- Absences greater than `30` trigger auto-fail.
- If the maximum achievable grade is below `50`, the course is considered non-recoverable.
- Suggestions are recalculated after relevant syllabus, week, or exam updates.

### Qualitative and Product-Level Findings

The implemented platform addresses three operational needs:

- early identification of at-risk students before final failure
- interpretable risk outputs through feature-importance explanations
- actionable intervention support through contextual AI suggestions and what-if simulation

The architecture supports both pedagogical inspection and operational scalability because prediction, storage, and user interaction are cleanly separated across services.
